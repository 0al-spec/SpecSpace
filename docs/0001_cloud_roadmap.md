# Анализ: менеджмент пользовательских workspace в ContextBuilder/SpecGraph

## Контекст и постановка задачи

Сервис обслуживает работу с большими спецификационными системами. Каждая система = workspace = набор файлов (`spec_dir`, `runs/`, `dialog_dir`) + конфигурация (capabilities, supervisor binary). Workspace должны быть изолированы друг от друга, но связаны через файловый протокол SpecPM (export/import compiled artifacts). Текущая реализация: один процесс = один workspace, фиксированный на старте.

Вопрос: какой архитектурный подход выбрать для облачного развертывания.

---

## Рассмотренные варианты

### Вариант A. Instance-per-workspace, постоянно работающие инстансы

Каждый workspace — отдельный процесс `server.py`, всегда запущен, имеет свой порт/URL. Workspace Manager — тонкий каталог, знающий список инстансов и проксирующий/редиректящий на нужный.

**За:**
- Изоляция = граница процесса (kernel-level), не приложения. Утечка состояния между WS невозможна по построению.
- SpecWatcher, capability checks, кэши — естественно per-process, без дополнительной логики.
- Существующий код `server.py` не требует изменений. Нулевой рефакторинг-риск.
- Сбой/OOM в одном workspace не затрагивает соседей.
- Resource limits (cgroups, ulimit) применяются прямо к процессу WS.
- Cold-start = 0: всё уже горячее.

**Против:**
- Линейный рост памяти: ~30–50 MB Python interpreter × N workspaces. 100 WS = 3–5 GB только baseline.
- Линейный рост портов/процессов: операционная нагрузка на supervision (systemd/k8s/docker).
- Платим за idle WS — пользователь редко работает во всех одновременно.
- Логи и метрики разбросаны по N процессам → нужна aggregation layer.

### Вариант B. Multi-tenant, единый процесс

Один `server.py` обслуживает все workspace. `workspace_id` пробрасывается через URL (`/ws/:id/api/...`) или header. Состояние workspace хранится в реестре, capabilities/spec_dir/dialog_dir resolved per-request.

**За:**
- Одна точка входа, один процесс — минимальная операционная поверхность.
- Память: единый interpreter + per-WS data, экономия на пустых WS.
- Нет cold-start.
- Логи/метрики централизованы.

**Против:**
- Большой рефакторинг: `workspace_id` через все 70+ handlers, capability flags становятся per-tenant lookup, SpecWatcher должен поддерживать N деревьев.
- Изоляция — application-level: баг в одном handler потенциально утекает между tenants. Аудит безопасности дороже.
- Blast radius глобальный: OOM/crash убивает всех пользователей.
- Кэш `_WORKSPACE_CACHES` уже есть, но многие места предполагают единый workspace — нужна полная ревизия.
- GIL-bound: тяжёлые операции одного WS блокируют остальных (если не уйти на async/процессы воркеров).

### Вариант C. Manager + on-demand instances (warm pool)

Workspace Manager запускает инстанс по требованию, держит idle-таймаут, останавливает неактивные. Активные WS — свои процессы; неактивные — на диске, без процесса.

**За:**
- Pay-per-use: платим только за активные WS. Подходит модели "у пользователя 50 проектов, активно работает в 3".
- Изоляция сохранена (как в A).
- Масштабируется до большого количества WS без линейного роста памяти.
- Естественная модель для serverless/k8s autoscaling.

**Против:**
- Cold-start latency: первый запрос после простоя = запуск процесса + индексация SpecWatcher (для большого `spec_dir` это секунды).
- Manager становится critical path — его падение отключает всех. Нужна HA-история.
- Больше движущихся частей: lifecycle (start/stop/health), state (idle timer), routing.
- "Тёплые" данные (кэш workspace listing) теряются при остановке.
- Сложнее отладка: проблема может быть в Manager, в инстансе, или в их взаимодействии.

### Вариант D. Self-host, без Manager

Пользователь сам запускает по `server.py` на нужный workspace. Каждый инстанс — отдельный URL (или порт на одном хосте). Управление списком — на стороне пользователя (закладки, скрипты).

**За:**
- Zero infra: ничего не строим сверх того что есть.
- Подходит для разработки и небольших команд.
- Полная транспарентность.

**Против:**
- UX страдает на >5 workspace.
- Нет централизованного контроля доступа.
- Не масштабируется на multi-user.

## Сравнительная таблица

| Критерий | A: Always-on | B: Multi-tenant | C: On-demand | D: Self-host |
|---|---|---|---|---|
| Рефакторинг кода | нулевой | большой | малый (Manager) | нулевой |
| Изоляция | kernel | app | kernel | kernel |
| Память (idle WS) | высокая | низкая | нулевая | высокая |
| Cold-start | нет | нет | секунды | нет |
| Blast radius сбоя | один WS | все | один WS + Manager | один WS |
| Операционная сложность | средняя | низкая (1 процесс) | высокая | низкая |
| Подходит для cloud SaaS | условно | да | да | нет |
| Подходит для self-host | да | избыточно | избыточно | да |

## Эксплуатационные особенности (любого варианта с Manager)

**Хранилище.** Локальная ФС работает только в self-host. Для cloud — общий volume (k8s PVC, EFS) или объектное хранилище с локальным кэшем. SpecWatcher поверх S3 не работает напрямую — нужен fuse-mount или периодический pull.

**Аутентификация.** Должна жить в Manager (single sign-on), а не в каждом инстансе. Инстансы доверяют запросам от Manager (mTLS или подписанные токены). Иначе придётся настраивать auth N раз.

**Логи.** Per-WS log streams со структурированным `workspace_id`. Aggregation (Loki/CloudWatch). Без этого debug в проде невозможен.

**Backup.** Естественная единица бэкапа = workspace directory. Snapshot на уровне ФС/PV. Cross-workspace атомарность не нужна — это и есть смысл изоляции.

**Resource quotas.** Per-WS лимиты CPU/memory/disk (cgroups в варианте A/C). В варианте B — только application-level (request limits, slow-query timeouts), что слабее.

**Cross-workspace SpecPM.** Файловый export/import работает в любой архитектуре — это правильная граница связности. Manager может предоставить UI для "загрузить compiled.md из WS-A в WS-B" через временное staging-хранилище.

**Idle shutdown.** В A не нужен; в C — критическая часть экономики (без него теряется смысл варианта).

**Observability.** Healthcheck endpoint в `server.py` уже есть неявно. Manager должен периодически опрашивать инстансы и удалять зомби. Метрики: per-WS request rate, latency, SpecWatcher reload count, capability availability.

## Рекомендация

**Краткосрочно (self-host, single user, до ~10 WS):** Вариант D. Никаких изменений, только документация.

**Среднесрочно (multi-user cloud, до ~100 WS на пользователя):** Вариант A + Workspace Manager. Минимальный рефакторинг (Manager — отдельный сервис), kernel-level изоляция, понятная операционка. Идеально для k8s: один Pod = один WS, Manager = один Deployment.

**Долгосрочно (large-scale SaaS, тысячи WS, разреженная активность):** Вариант C. Но входить в него только когда экономика A становится больно — преждевременная оптимизация даст cold-start headache без пользы.

**Вариант B исключён.** Затраты на рефакторинг и риски потери изоляции не окупаются экономией памяти. Для домена "разработка спецификаций" изоляция — это feature, не overhead.

## Главные риски и митигации

| Риск | Вариант | Митигация |
|---|---|---|
| Линейный рост памяти | A | Resource limits per-WS; в проде мониторить, выбраковывать неактивные >7 дней |
| Cold-start UX | C | Прогрев топ-N WS пользователя при логине; индикатор "starting…" в UI |
| Manager как SPOF | A/C | Manager без состояния (state в БД); active-passive replicas |
| Drift между WS-инстансами разных версий | A/C | Rolling deploy с health check; Manager знает версию каждого инстанса |
| Утечка между tenants | B | Только если выбран B — security review всех handlers, fuzz testing routing |

---

## Вывод

Текущая модель (instance-per-workspace) не является техдолгом — это осознанная граница изоляции, соответствующая природе домена. Проблема не в архитектуре сервера, а в отсутствии слоя над ним. Workspace Manager как отдельный лёгкий сервис закрывает разрыв между текущей реализацией и облачной моделью без необходимости менять `server.py`. Вариант B (multi-tenant) — соблазнительный по экономике памяти, но требует переписывания фундамента ради количественного выигрыша, который не транслируется в качественные преимущества для пользователя.
